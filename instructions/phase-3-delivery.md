## Правила декомпозиции задач

Каждая задача должна:

- быть атомарной;
- давать один чёткий результат;
- содержать критерии завершения;
- содержать зависимости, если они есть;
- не объединять несвязанные цели.

Новые задачи начинают в статусе `Backlog`.

Агент не создаёт задачи на выполнение до тех пор, пока результаты сбора требований не достаточны для Definition of Ready.

---

## Правила работы с GitHub Issues и GitHub Project

- Новые задачи попадают в `Backlog`.
- Взятая в работу задача переходит в `In Progress`.
- Завершённая задача закрывается: статус `Closed`.
- Агент поддерживает доску в актуальном состоянии.
- Агент создаёт и обслуживает метки.

Минимальный набор меток:

- `type: epic`
- `type: feature`
- `type: bug`
- `type: task`
- `area: frontend`
- `area: backend`
- `area: infra`
- `area: docs`
- `area: data`
- `priority: high`
- `priority: medium`
- `priority: low`
- `status: blocked`
- `status: needs-info`

---

## Правила работы с внешними библиотеками

Перед работой с внешней библиотекой, фреймворком, SDK или внутренней общей библиотекой — не полагайся на память или предположения.

Порядок проверки:

1. Проверь локальную документацию в репозитории.
2. Проверь существующие примеры использования в кодовой базе.
3. Проверь подключённый официальный источник документации.
4. Если подтверждение не найдено — явно обозначь неопределённость и не изобретай решение.

## Правила работы с pull request

Pull request включены. Поток PR ограничен одной задачей.

Перед первым коммитом реализации для каждой задачи:

1. Классифицируй задачу: требует ли она PR по политике репозитория.
2. Явно зафиксируй режим выполнения перед первым коммитом реализации.
3. Если PR требуется: создай ветку задачи или переключись на неё. Не пушь коммиты реализации напрямую в `main`.
4. Если PR не требуется: зафикcируй это явно и обоснуй относительно политики репозитория.

**Политика создания PR: `{{pull_requests.creation_mode}}`**

<!-- IF:pull_requests.creation_mode=for_every_task -->
Каждая задача использует собственную ветку и PR.
<!-- END IF -->
<!-- IF:pull_requests.creation_mode=for_significant_tasks -->
PR требуется только для значимых задач. Значимые задачи — те, которые меняют код приложения или инфраструктуры, изменяют поведение системы или затрагивают архитектуру, API, безопасность, миграции или внешние интеграции. Задачи только с документацией и мелкие технические изменения не являются значимыми, если репозиторий не определяет иное.
<!-- END IF -->
<!-- IF:pull_requests.creation_mode=manual_per_task -->
Необходимость PR для задачи определяется явным решением человека для каждой задачи. Не решай молча — запроси или получи явное решение человека прежде чем считать задачу свободной от PR.
<!-- END IF -->

**Политика проверки:**

<!-- IF:pull_requests.review.required=false -->
Проверка не требуется. PR может быть слит, когда все остальные условия слияния выполнены.
<!-- END IF -->
<!-- IF:pull_requests.review.required=true -->
Проверка обязательна. Не считай задачу готовой к слиянию до выполнения требований проверки.
<!-- END IF -->

<!-- IF:pull_requests.review.reviewers=human -->
Требуется проверка человеком. Самопроверки агента недостаточно.
<!-- END IF -->
<!-- IF:pull_requests.review.reviewers=ai -->
Проверка агентом — обязательный путь проверки. Зафиксируй результат проверки в PR в виде сводки или комментария перед слиянием.
<!-- END IF -->
<!-- IF:pull_requests.review.reviewers=both -->
Требуется и проверка агентом, и проверка человеком. Зафиксируй результат проверки агентом в PR до начала проверки человеком.
<!-- END IF -->

<!-- IF:pull_requests.review.agent_must_read_comments=true -->
Прочитай все комментарии и сводки проверки PR перед завершением обработки проверки.
<!-- END IF -->
<!-- IF:pull_requests.review.agent_must_reply_to_comments=true -->
Отвечай на комментарии PR там, где процесс ожидает прямого ответа.
<!-- END IF -->
<!-- IF:pull_requests.review.agent_must_apply_accepted_feedback=true -->
Применяй принятые замечания по проверке до того, как считать PR готовым к слиянию.
<!-- END IF -->

**Политика слияния:**

<!-- IF:pull_requests.merge.squash_commits=true -->
Коммиты задачи объединяются в один перед слиянием.
<!-- END IF -->
<!-- IF:pull_requests.merge.squash_commits=false -->
Коммиты задачи сохраняются по отдельности при слиянии.
<!-- END IF -->
<!-- IF:pull_requests.merge.integration_method=merge -->
Интеграция ветки выполняется через merge commit.
<!-- END IF -->
<!-- IF:pull_requests.merge.integration_method=rebase -->
Интеграция ветки выполняется через rebase.
<!-- END IF -->
<!-- IF:pull_requests.merge.require_green_checks=true -->
Не считай PR готовым к слиянию, пока обязательные проверки не прошли или отсутствуют.
<!-- END IF -->
<!-- IF:pull_requests.merge.min_approvals!=0 -->
Перед слиянием требуется не менее `{{pull_requests.merge.min_approvals}}` подтверждений.
<!-- END IF -->
<!-- IF:pull_requests.merge.allow_agent_self_merge=false -->
Агент не должен выполнять слияние PR. Остановись перед слиянием и дождись уполномоченного участника.
<!-- END IF -->
<!-- IF:pull_requests.merge.allow_agent_self_merge=true -->
Агент может выполнить слияние PR, когда все настроенные условия выполнены.
<!-- END IF -->
<!-- IF:pull_requests.merge.agent_configure_branch_protection=true -->
Перед началом любой работы по реализации настрой защиту ветки на основной ветке через GitHub API. Обязательные правила: требовать pull request перед слиянием, установить минимальное количество подтверждений согласно `pull_requests.merge.min_approvals`, отключить обход защиты. Требуется административный доступ к репозиторию.
<!-- END IF -->

## Карта проекта

`docs/project-map.md` — актуальная карта файловой структуры проекта. Карта должна содержать дерево файлов с кратким описанием назначения каждой папки и ключевого файла.

## Правила документирования

- Любое значимое решение должно быть зафиксировано в `docs/`.
- Изменения требований должны обновлять `docs/02-business-requirements.md`.
- Изменения архитектуры должны обновлять `docs/05-architecture.md` и `docs/06-decision-log.md`.
- Изменения стека должны обновлять `docs/04-tech-stack.md`.
- Изменения рабочего процесса должны обновлять `docs/07-workflow.md`.
- Изменения политики рабочего процесса должны обновлять `.ai-dev-template.config.json` и `docs/11-workflow-configuration.md`, если меняется смысл.

## Definition of Ready

Задача готова к выполнению только тогда, когда ясны:

- цель;
- входные данные;
- границы;
- зависимости;
- критерии завершения;
- ожидаемый результат.

---

## Definition of Done

Задача считается выполненной только когда:

- требуемый результат существует;
- документация обновлена;
- состояние Issue обновлено;
- статус в GitHub Project обновлён;
- важные решения зафиксированы;
- новые риски или ограничения зафиксированы;
- создан отдельный git-коммит для этой задачи.

<!-- IF:pull_requests.enabled=true -->
Завершение задачи также требует соответствия настроенной политике PR, проверки и слияния.
<!-- END IF -->

Если в проекте нет специфического стандарта коммитов — используй Conventional Commits и включай ссылку на Issue в заголовок или тело коммита. Язык сообщения коммита должен соответствовать `.ai-dev-template.config.json`.

---

## Порядок завершения каждой задачи

После каждой выполненной задачи агент должен выполнить по порядку:

1. Зафиксировать результат в канонических источниках.
2. Обновить документацию.
3. Обновить связанный Issue.
4. Обновить карточку в GitHub Project.
5. Обновить метки или статус если нужно.
<!-- IF:project_map.enabled=true -->
6. Если структура проекта изменилась — обновить `docs/project-map.md`.
<!-- END IF -->
7. Создать отдельный коммит для задачи.
8. Убедиться что никакое значимое состояние не существует только в контексте сессии.
9. Только после этого сбросить или очистить рабочий контекст.

<!-- IF:pull_requests.enabled=true -->
При наличии PR: завершение задачи включает также прохождение полного цикла проверки и слияния согласно политике репозитория.
<!-- END IF -->
